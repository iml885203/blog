# 技術博客寫作風格指南

本指南基於 Dan Abramov (overreacted.io) 和阮一峰的寫作風格，旨在創作有溫度、有深度、易於閱讀的技術文章。

---

## ⚖️ 核心精神：平衡的藝術

**最重要的原則：避免過度矯正**

技術文章需要在三者之間找到平衡：

```
故事性（建立連結）────────── 系統性（實用價值）
          \                    /
           \                  /
            \                /
             \              /
              \            /
               \          /
                \        /
                 \      /
                  \    /
                   \  /
              技術深度（專業性）
```

**理想比例：**
- 40% 故事和案例（吸引讀者、建立信任）
- 40% 系統性內容（實用建議、可執行步驟）
- 20% 技術深度（原理解釋、專業洞察）

**常見錯誤：**
- ❌ 100% 列表 → 像 PPT，無聊
- ❌ 100% 故事 → 像散文，冗長
- ❌ 100% 技術 → 像論文，難懂

---

## 核心原則

### 1. 用故事開場，但要精簡

❌ **過於平淡：**
```markdown
在 AI 快速發展的時代，開發工具也開始融入 AI 的能力...
```

✅ **吸引人但不誇張：**
```markdown
上個月我讓 AI 生成了一段程式碼。看起來很完美，
但跑測試時失敗了一堆。那一刻我開始思考...
```

⚠️ **過於戲劇化（避免）：**
```markdown
凌晨三點，我盯著螢幕上的 500 行程式碼，
心跳加速，手心冒汗...（太像小說了）
```

**故事長度建議：**
- 開頭故事：100-150 字
- 主體故事：50-100 字
- 總故事數：3-4 個就好

### 2. 具體但不過度

✅ **適度具體：**
- 「上週」「兩個月前」
- 「一堆測試失敗」「幾個 edge cases」
- 「花了一整天」

❌ **過度具體（反而不真實）：**
- 「上週三凌晨 2:47 分」
- 「精確 27 個測試失敗」
- 「花了 8 小時 23 分鐘」

### 3. 保留系統性結構

技術文章需要讓讀者能快速掃描和查找。

✅ **好的平衡：**
```markdown
## 減少 AI 幻覺的方法

上個月我踩了個坑：AI 生成的程式碼看起來對，
實際上完全誤解了需求...（50字故事）

從這次經驗，我整理出三個關鍵做法：

### 1. 寫詳細的 README
當 AI 能讀到「這個模組負責什麼」「有哪些限制」，
生成品質會好很多。

（配 50 字的實際例子）

### 2. 設定專案規範
### 3. 強制 Code Review

（每個都配短例子）
```

**關鍵：** 用故事引入主題，但核心內容要結構化。

### 4. 失敗與成功的比例

❌ **全是成功 → 不可信**
❌ **全是失敗 → 沒權威**

✅ **理想比例：**
- 1-2 個失敗經歷（建立真實感）
- 3-4 個成功經驗（展示方法有效）
- 重點在「從失敗學到什麼」

## 語言風格

### ✅ 使用這些表達

**真實感：**
- 老實說、我發現、後來才明白
- 試了幾次、第一次失敗了
- 上週、兩個月前、那時候

**對話感：**
- 你可能會問、想像一下
- 聽起來簡單對吧？但是...

**不確定性：**
- 我不確定這是最好的做法
- 在我的情況可行，但你的可能不同
- 可能是我理解錯了

### ❌ 避免這些表達

**AI 感重：**
- 綜上所述、總而言之、毋庸置疑
- 在 XX 快速發展的時代
- 身為工程師，我深刻體會到
- 具體來說、反之、實例分享

**過度承諾：**
- 完美解決方案、最佳實踐
- 一定能、必然會、保證
- 全面優化、大幅提升（除非有數字）

---

## 文章結構

### 開頭（100-150 字）

**三種有效方式：**
1. **場景開始**：上個月我遇到...
2. **問題開始**：為什麼 X 總是...
3. **觀察開始**：用了半年 X 後，我發現...

**避免：**
- ❌ 從定義開始
- ❌ 過度戲劇化（凌晨、心跳、冒汗）
- ❌ 「在 XX 時代」的開場白

### 主體結構

**推薦流程：**
```
1. 開頭故事（100字）→ 引出主題
2. 核心觀點（系統化列表或段落）
3. 實例支撐（2-3個50字小故事）
4. 深入原理（可選，視讀者程度）
5. 實踐建議（結構化列表）
```

**關鍵：** 故事和列表交替，不要連續超過 3 個段落都是同一種形式。

### 結尾

✅ **好的結尾：**
- 簡短總結核心觀點（1-2 句）
- 承認局限性
- 開放式提問或邀請討論

❌ **避免：**
- 「綜上所述，我們討論了 A、B、C...」
- 重複前面說過的內容
- 「希望對大家有所幫助」

## 比喻的使用

### ✅ 好的比喻

**特點：**
- 貼近生活經驗
- 簡化但不失真
- 之後補充技術細節

**範例：**
```markdown
LLM 就像一個記憶超強但偶爾會瞎掰的朋友。
他讀過所有文件，但回答問題時會受運氣影響。

更技術地說，LLM 是機率模型。每次生成時都從機率分佈
中採樣，而這個分佈受 prompt、temperature、
訓練資料等因素影響...
```

### ❌ 過度簡化

**問題：**
- 只有比喻，沒有技術解釋
- 比喻不夠準確，專業讀者會質疑

**修正：** 比喻後要補充「更準確地說...」

---

## 篇幅控制

**建議長度：**
- 短文：1,500-2,000 字（單一主題）
- 中文：2,500-3,500 字（多個主題）
- 長文：不超過 4,000 字

**超過 3,500 字時的檢查：**
- 是否有次要故事可以刪除？
- 是否有重複說明的地方？
- 是否可以拆成兩篇文章？

## 視覺元素的使用

### 何時使用圖片、圖表、範例

**原則：輔助理解，不要裝飾**

✅ **應該使用的情況：**
- 文字難以描述的流程（用流程圖）
- 對比複雜的數據（用表格）
- 介面操作說明（用截圖）
- 抽象概念視覺化（用示意圖）
- 程式碼執行結果（用範例輸出）

❌ **不需要的情況：**
- 純粹為了「好看」
- 重複文字已經說明的內容
- 過於簡單的概念（如「點擊按鈕」）

### 標記需要配圖的位置

在寫作過程中，用註解標記「需要配圖」：

```markdown
## 如何使用 Cursor

<!-- 📸 TODO: 需要截圖 - Cursor 主介面，標示出 AI chat 位置 -->

當你打開 Cursor 後，右側會看到 AI 對話框...

<!-- 📊 TODO: 需要流程圖 - 從輸入 prompt 到程式碼生成的流程 -->

整個流程是這樣的：首先你輸入需求...

<!-- 💻 TODO: 需要程式碼範例 - before/after 對比 -->

重構前後的差異很明顯...
```

**標記符號：**
- 📸 = 需要截圖
- 📊 = 需要圖表/流程圖
- 💻 = 需要程式碼範例
- 🎨 = 需要示意圖

### 圖片類型和使用場景

#### 1. 截圖（用於操作說明）

**何時用：**
- 介紹工具介面
- 展示設定步驟
- 說明操作流程

**注意：**
- 標註重點區域（圈起來、箭頭指示）
- 避免截圖太大（裁切到關鍵部分）
- 加上簡短的圖說

**範例：**
```markdown
<!-- 📸 TODO: Windsurf Flow mode 截圖，圈出 AI 建議的彈出框 -->
![Windsurf Flow Mode](windsurf-flow.png)
*Flow mode 會主動彈出建議（紅框處）*
```

#### 2. 流程圖（用於說明流程）

**何時用：**
- 多步驟流程
- 決策樹
- 系統架構

**工具建議：**
- Excalidraw（手繪風格，輕鬆自然）
- Mermaid（程式碼生成，可維護）
- draw.io（專業圖表）

**範例：**
```markdown
<!-- 📊 TODO: 流程圖 - AI 生成程式碼的完整流程 -->

\`\`\`mermaid
graph TD
    A[輸入需求] --> B[AI 生成初稿]
    B --> C{測試通過?}
    C -->|是| D[提交程式碼]
    C -->|否| E[人工 debug]
    E --> B
\`\`\`
```

#### 3. 對比表格（用於比較）

**何時用：**
- 工具比較（如 Cursor vs Windsurf）
- 優缺點對比
- 版本差異

**注意：**
- 不超過 5 個比較項目
- 每格內容精簡（< 30 字）
- 加上表格說明

**範例：**
```markdown
| 情境 | 建議工具 | 理由 |
|------|---------|------|
| 快速迭代 | Windsurf | AI 主動建議 |
| 精確控制 | Cursor | 手動觸發 |
| 學習新技術 | Cursor | 逐步引導 |
```

#### 4. 程式碼範例（用於技術說明）

**何時用：**
- 重構前後對比
- 常見錯誤示範
- API 使用範例

**格式：**
```markdown
<!-- 💻 TODO: 程式碼範例 - AI 生成的錯誤 vs 正確寫法 -->

❌ **AI 第一次生成的版本：**
\`\`\`python
def process_data(data):
    return data.filter(lambda x: x > 0)  # 問題：data 可能是 None
\`\`\`

✅ **修正後：**
\`\`\`python
def process_data(data):
    if data is None:
        return []
    return [x for x in data if x > 0]
\`\`\`

關鍵差異：加上了 None check，避免程式崩潰。
```

### 節奏控制

**理想密度：**
- 每 500-800 字配一個視覺元素
- 不要連續兩個段落都有圖片
- 長文（>3000 字）：4-6 個視覺元素
- 短文（<2000 字）：2-3 個視覺元素

**錯誤的節奏：**
```markdown
（文字 100 字）
（圖片 1）
（文字 50 字）
（圖片 2）
（文字 80 字）
（圖片 3）
← 太密集，像教科書
```

**良好的節奏：**
```markdown
（開頭故事 200 字）
（核心概念說明 400 字）
（圖片 1：概念示意圖）
（實作步驟 300 字）
（程式碼範例）
（經驗分享 400 字）
（圖片 2：結果截圖）
← 舒服，有呼吸感
```

### 視覺元素檢查清單

寫完文章後檢查：

- [ ] 每個圖片/圖表都有明確目的（不只是裝飾）
- [ ] 有圖說或前後文說明
- [ ] 視覺元素間距合理（不會太密集）
- [ ] 截圖有標註重點（圈出、箭頭、文字說明）
- [ ] 程式碼範例有註解說明關鍵部分
- [ ] 表格不超過 5 列 × 4 欄（太大會難讀）
- [ ] 流程圖節點不超過 10 個（太複雜要拆分）

### 常見錯誤

❌ **過度使用：**
每個段落都配圖，讓讀者視覺疲勞

❌ **圖文不符：**
圖片和文字說的不一致

❌ **圖片過大：**
一張截圖佔滿整個螢幕，要滑很久才能繼續讀

❌ **缺乏說明：**
丟一張圖，沒說這張圖要看什麼

### 實際操作建議

**寫作流程：**

1. **第一遍：只寫文字**
   專注內容，不要被圖片打斷思路

2. **第二遍：標記需要配圖的地方**
   用 `<!-- 📸 TODO -->` 標記

3. **第三遍：製作/尋找圖片**
   - 截圖：自己操作軟體截圖
   - 流程圖：用 Mermaid 或 Excalidraw
   - 示意圖：用 Excalidraw 手繪
   - 程式碼：實際執行確保正確

4. **第四遍：檢查節奏**
   確保視覺元素分佈均勻

## 快速檢查清單

發布前快速檢查（5 分鐘）：

### 內容平衡
- [ ] 故事占比 30-50%（不超過 60%）
- [ ] 有系統性的列表或步驟
- [ ] 至少 1 個失敗或挫折經歷
- [ ] 說明了「為什麼」而非只有「怎麼做」

### 語言風格
- [ ] 無 AI 感詞彙（綜上所述、具體來說）
- [ ] 有具體但不過度的時間描述
- [ ] 句子長短有變化
- [ ] 避免過度承諾（完美、最佳、一定）

### 結構
- [ ] 開頭有吸引力（100-150 字）
- [ ] 核心內容結構化（能快速掃描）
- [ ] 結尾留下思考空間
- [ ] 總字數在合理範圍（< 4,000）

## 一個完整範例

這是平衡良好的段落結構：

```markdown
## 為什麼 AI 有時候會瞎掰

（故事 - 50字）
上個月我問 AI 一個 API 用法，它很自信地給了範例。
我照著做，結果報錯：這個方法根本不存在。

（解釋 - 100字）
這就是 AI 的「幻覺」問題。LLM 是機率模型，它會根據
訓練資料預測「最可能的下一個詞」，而不是查證事實。
當它不確定時，不會說「我不知道」，而是編造一個
聽起來合理的答案。

（系統性建議 - 150字）
要減少幻覺，我現在會這樣做：

1. **提供具體 context**
   不只問「怎麼用」，而是貼上相關文件或程式碼

2. **交叉驗證**
   重要的東西一定查官方文件確認

3. **要求引用來源**
   問 AI「這個資訊來自哪裡？」

（小故事收尾 - 30字）
從那次開始，我把 AI 當成「快速草稿工具」，
而不是「權威答案來源」。
```

**分析：**
- 故事：50 + 30 = 80 字（35%）
- 解釋：100 字（43%）
- 建議：50 字（22%）
- 總計：230 字，平衡良好

## Dan Abramov 的真正風格

很多人誤解 Dan Abramov 的風格是「寫很多故事」。

**實際上他的文章：**
- 用故事引入主題（10-15%）
- 深入技術討論（60-70%）
- 程式碼範例和原理解釋（15-20%）

**關鍵差異：**
他的故事是「引子」，不是「主體」。真正的內容是深入的技術洞察。

**範例：** [Why Do React Hooks Rely on Call Order?](https://overreacted.io/why-do-hooks-rely-on-call-order/)
- 開頭：短故事（為什麼 Hooks 設計成這樣？）
- 主體：深入討論各種替代方案的優缺點
- 結尾：總結設計取捨

**我們應該學習的：**
- ✅ 用故事引發興趣
- ✅ 保持技術深度
- ✅ 誠實面對局限
- ❌ 不是寫成技術散文

## 最後提醒

### 三個最容易犯的錯誤

1. **矯枉過正**
   從「全是列表」變成「全是故事」

2. **過度具體**
   精確到「凌晨 2:47 分」反而不真實

3. **失去實用性**
   讀者看完覺得「寫得很好，但我能幹嘛？」

### 記住

好的技術文章讓讀者：
- **想讀**（故事吸引人）
- **能懂**（解釋清楚）
- **會用**（有實踐步驟）

這三者缺一不可。
